1. 加载申请

并行读取证券详情、当前池、申请日志、步骤和附件。

2. 校验待办

确认步骤仍为 pending 且当前用户是实际处理人。

3. 记录动作

写入同意、驳回、修改重提、意见、附件和处理时间。

4. 推进流程

按会签/抢占模式及条件边创建下一节点待办。

5. 结束生效

通过时更新最终池状态；驳回或终止只结束申请。

查询、校验与操作逻辑

按当前前后端实现归纳主要触发场景。

页面加载

根据 URL 中的业务代码和 adjustBatchNo 定位申请，读取详情、池状态、日志、步骤与附件。页面展示的是申请快照及当前状态，不把历史日志当作最终在池事实。

并发与权限

提交前重新查询当前步骤；只有 pending 状态和合法处理人才能更新。Mapper 采用状态条件更新防止重复审批，管理员也必须遵循步骤状态。

会签与抢占

会签节点等待配置人员全部处理；抢占节点由首位处理人完成后，其余 pending 步骤标记 skipped。之后依据动作和条件边寻找下一节点。

修改后重提

进入修改节点时重新调用 checkAdjust 校验池状态、关系和流程；可增删附件并重新绑定申请。普通详情页的首次提交不走这一审批入口。

流程结束

最终通过时对 ip_pool_status 执行调入新增或调出逻辑删除；驳回不改变当前在池状态。

附件与报告

审批附件写 sys_attachment；最终入池通过且满足条件时生成 rrs_report_in，并将相关附件复制到报告关联。

关键规则与边界

帮助区分当前状态、历史记录和流程实例。

乐观并发

当前步骤必须仍是 pending；已处理步骤不能再次提交。

状态机

00 待审核、10 待审批、11 驳回待修改、20 通过、21 驳回、32 自动审批、99 撤回；只有通过类结果落最终池状态。

流程快照

审批读取申请锁定的 flowVersionId，不随流程定义后来发布的新版本漂移。
