总体结论

当前证券池调整主链路已经能够从“查证券 → 查当前池 → 校验 → 发起 → 审核 → 生效 → 查询/历史/待办回看”形成闭环；真正需要关注的是边界条件、权限口径、流程配置口径和数据量增长后的性能兜底。

结论

主链路基本合理

单笔调库和批量调库都落到同一套 ip_adjust_log、ip_adjust_step、ip_pool_status 语义，审批通过后再写当前池状态，历史与查询也能回看。

业务确认

批量提交是整批原子

任意证券失败会整体回滚。这对数据一致性好，但如果业务希望“部分成功”，需要明确改造。

主要风险

流程语义依赖节点名称

发起、复核、修改、审批、O32 自动审批等判断使用节点中文关键字，流程配置改名后可能影响状态流转。

优先补充

索引、幂等与巡检兜底

当前关键运行表查询索引偏少，唯一性也主要靠业务代码维护。建议优先补业务锁/幂等键/巡检，再按实际 SQL 补组合索引。

页面职责边界：详情页、审核页不要混用

这是本页最容易误判的点：详情页不是审批处理页，驳回待修改也不是在详情页重新提交。

详情页

security_pool_adjust_detail.html

  • entryMode=view：只读展示证券信息、当前所在池、调库记录、流程步骤和附件。
  • entryMode=adjust：首次调库申请，走 checkAdjust + addAdjustLogWithFiles，会新建 ip_adjust_log。
  • entryMode=process 在详情页会归一为只读，不会处理审批动作。
  • 终态记录可回看、附件可下载，但没有审批按钮。
审核页

security_pool_adjust_approve.html

  • 我的事宜待处理入口跳转到审核页，携带 adjustLogId、adjustBatchNo 和 entryMode=process。
  • 复核/审批/修改节点都走 securityPoolAdjustFlow/submitAdjustAudit。
  • audit_status='11' 的“驳回待修改”在这里重新提交，更新已有步骤和日志，不新建调库记录。
  • 修改节点 approve 才允许通过 attachmentChanges 增量变更附件；reject 表示终止流程。
体检结论：如果后续排查“为什么详情页不能处理待办”，不要改详情页，应检查我的事宜跳转是否进入审核页，以及当前用户是否命中 currentPendingStep。

状态机与真实运行口径

调库状态由申请日志状态和流程步骤状态共同表达；只有最终通过才改变当前池。

-1无效调整，通常表示校验失败或操作中止。
00已提交待审核；首次提交或修改节点重新提交。
10复核通过待审批；仍不影响当前池。
11复核驳回待修改；发起人在修改节点处理。
20审批通过；调入写入当前池，调出软删当前池。
21审批驳回；流程结束，不改变当前池。
32O32 自动审批预留；当前运行通常最终仍落为 20。
99修改节点终止流程，视为发起人撤回。
触发动作 日志状态变化 步骤状态变化 是否影响 ip_pool_status
非直通提交 00 start 自动完成，发起节点 submit，下一审批节点 pending 否
直通提交 20 start / initiator / end 自动走完 是，立即入池或软删出池
复核驳回 11 复核节点 reject，创建修改节点 pending 否
修改节点提交 00 修改节点 submit，重新回到复核/后续节点 否
审批通过 20 审批节点 approve，自动节点/结束节点继续推进 是，批次内全部调库项统一生效
审批驳回 / 修改终止 21 / 99 沿 reject 路径创建结束节点步骤 否

当前比较合理的设计

这些点建议保留，不要为了“看起来简单”反而破坏链路一致性。

合理

当前状态与申请日志分离

ip_adjust_log 记录过程，ip_pool_status 只承载最终已生效状态，查询和历史口径清晰。

合理

批次号贯穿日志和步骤

联动、互斥、同组调库使用 adjust_batch_no 串起来，详情和审批可以按批次展示。

合理

批量复用单笔规则

批量调整没有复制一套规则，而是按证券调用单笔校验/提交，减少规则漂移。

合理

提交前有多层校验

到期、进行中、重复入池、容量、来源池、限制池、互斥、弹性禁投池都有覆盖。

合理

会签/抢占处理有区分

会签必须当前节点全部处理完才推进；非会签会跳过同节点其他待处理人，符合常见审批模型。

合理

审批通过后沉淀报告

手工上传的信评报告在流程结束后进入内部报告库，有利于形成资料闭环。

合理

发起人与审批人职责隔离

后端会校验发起人不能处理后续审批节点；发起/修改节点除外，管理员代办除外，避免自提自批。

合理

管理员代办有明确口径

currentUserId='1001' 被视为管理员，可见全部待办并可代办；代办他人步骤时处理意见会追加管理员操作说明。

合理

详情与审批页面分工清晰

只读详情不负责流程处理；待办审批集中在审核页，能减少“详情页误改流程状态”的风险。

问题与风险点

以下不是全部都是 bug，其中一部分是“业务口径必须确认”，否则后面需求迭代容易产生理解偏差。

高优先级

流程语义依赖中文节点名

发起、复核、修改、审批、O32 自动审批通过 node_label/node_code/node_type 的文本关键字识别。

  • 如果流程配置人员把“复核”改成“审核”，状态可能不再进入 10 或 11 预期分支。
  • 建议后续在流程节点配置中增加稳定的业务语义字段。
高优先级

条件路由表已建但运行逻辑预留

wf_edge_cond_rule 与连线条件字段已经存在，但审批流转时条件匹配仍以预留逻辑为主。

  • 复杂分支不要误以为“配置了条件就一定生效”。
  • 当前更可靠的路由依据仍是 route_action 与处理动作。
高优先级

重复入池缺数据库兜底

校验阶段会查是否已在池，但表结构没有看到针对 security_code + target_pool_id + audit_status + is_deleted 的唯一性兜底。

  • 并发提交或异常重放可能产生多条有效在池记录。
  • 考虑当前项目弱物理约束风格，优先建议业务锁、幂等键或提交前二次校验。
高优先级

进行中判断只看 pending 步骤

当前“证券存在进行中流程”以 ip_adjust_step.step_status='pending' 为准。

  • 如果流程配置异常导致已提交日志没有 pending 步骤，可能放过新调库。
  • 建议把 ip_adjust_log.audit_status in ('00','10','11') 作为辅助兜底或巡检项。
需确认

权限口径不完全一致

调库池选择和批量目标池会校验可调权限；证券池查询和历史查询主要按前端传入的池树过滤，未看到同样的后端查看权限过滤。

  • 如果业务允许全员查看，则当前合理。
  • 如果需要按 viewable 权限隔离，需要后端统一补口径。
需确认

批量调整没有部分成功语义

当前批量提交外层事务包住全部证券，任意一只异常会整体回滚。

  • 数据一致性更好，但大批量场景用户会希望知道失败行。
  • 建议页面文案明确“整批提交、整批回滚”。
需确认

详情默认只看活跃流程

queryAdjustLogList 不传批次号时会过滤掉已通过、已驳回、已撤回等终态。

  • 作为“当前处理详情”合理。
  • 作为“历史详情”必须从历史页面带 adjustBatchNo 进入，否则会误以为没有记录。
需确认

O32 状态码当前更像预留

audit_status='32' 在字典中存在，但自动审批节点通常通过 auto_process 步骤推进，最终日志状态仍落 20。

  • 如果业务要单独展示 O32 中间状态，需要明确写入和回查逻辑。
  • 否则文案上应说明 32 是预留，不是当前稳定运行态。
需确认

我的事宜不是全量过程视图

我的事宜取每条调库日志的最新步骤，并用 pending/completed 区分待办已办。

  • 它适合做入口，不适合替代完整流程轨迹。
  • 同一批次多条日志时，展示口径需要保持“按日志”还是“按批次”一致。
需确认

弹性禁投池目前仍是硬阻断

代码注释提到弹性限制“可由特定角色绕过”，但当前调入/调出校验阶段直接加入失败原因。

  • 如果弹性禁投应能被审批放行，需补“例外申请/高级审批”链路。
  • 如果现在就是硬阻断，建议修改命名或说明，避免误解。
需确认

证券基础信息更新可能清空字段

调库提交时会用前端传入的可编辑字段覆盖证券信息，显式传入 null 会清空字段。

  • 当前前端若传完整表单则合理。
  • 后续若改成局部提交，要避免把未展示字段误置空。
需确认

批量独立选择页尚未打通

batch_security_pool_adjust.html 主页面链路完整；但独立选择页当前没有把选中证券回传主页面。

  • 如果要启用独立选择页，需要补 sessionStorage、postMessage 或 URL 回传机制。
  • 否则链路体检和验收应以批量主页面为准。
优化

关键表缺少业务索引

SQL 中关键表只看到主键，没有看到 security_code、adjust_batch_no、handler_id、target_pool_id 等高频查询索引。

  • 数据量上来后，查询、待办和审批流转会受到影响。
  • 建议补组合索引并根据实际 SQL 走 explain。
优化

我的事宜跳转页签体验不统一

当前页面内部使用 window.location.href 跳转，会在当前 iframe 内切换，不一定新增工作台页签。

  • 如果希望和首页快捷入口一致，应走父窗口打开页签能力。
  • 这属于体验问题，不影响后端业务闭环。

建议补充清单

按“先稳业务正确性，再补体验和性能”的顺序推进。

优先级 建议项 为什么要做 建议落点
P1 流程节点增加稳定业务语义 避免流程节点改名导致状态流转错判。 wf_flow_node 或节点审批配置增加语义字段；Service 改为读字段。
P1 补齐条件路由执行能力 当前条件规则表更像配置预留，复杂流程分支可能没有按条件真正路由。 SecurityPoolAdjustFlowService.matchesConditionRoute 与 wf_edge_cond_rule。
P1 增加重复入池幂等兜底 避免并发提交、重复请求产生多条有效状态。 优先业务锁/幂等键/提交前二次校验；如需唯一索引再结合项目弱物理约束规范评估。
P1 补进行中流程巡检口径 防止有 00/10/11 日志但没有 pending 步骤的异常状态漏拦。 校验阶段或定时巡检中增加异常识别。
P2 明确查询/历史查看权限 查看权限如果需要隔离，应后端统一控制，不能只靠前端池树。 证券池查询、历史查询、详情入口参数校验。
P2 明确 O32 状态是否需要落库 字典有 32，但当前自动节点通常最终写 20；展示和接口口径需统一。 状态字典、审核页展示、resolveProcessingNodeAuditStatus。
P2 明确批量是否允许部分成功 现在是整批回滚。业务如果希望“成功行提交”，需要重构事务边界和返回结构。 批量调整 Service 和前端结果提示。
P2 处理批量独立选择页去留 独立选择页尚未与主页面打通，容易形成“看得到但用不了”的入口。 删除入口、改成组件，或补 sessionStorage/postMessage 回传。
P3 给核心表补索引 待办、历史、查询和审批都依赖高频条件查询。 ip_adjust_log、ip_adjust_step、ip_pool_status。
建议验收时不要只测“能提交”。至少覆盖：直通调入、非直通调入、审批驳回待修改、修改后再提交、审批终止、调出、联动调入、互斥配套调出、批量整批回滚、我的事宜待办跳转、历史带批次进入详情。

验收矩阵：建议按场景逐条回归

这部分用于后续开发、迁移或演示前自检。重点不是“页面能打开”，而是每条链路的数据落点是否正确。

直通调入提交后 ip_adjust_log=20，同事务写 ip_pool_status。
非直通调入提交后日志为 00，只创建初始步骤，不改变当前池。
复核驳回日志变 11，创建修改节点 pending，不写当前池。
修改后提交审核页走 submitAdjustAudit，日志回到 00，不新建日志。
修改节点终止日志变 99，沿 reject 路径写结束步骤，不影响当前池。
审批通过日志变 20，批次内调入写入、调出软删 ip_pool_status。
审批驳回日志变 21，流程结束，不改当前池状态。
联动调入自动项与手工项共享 adjust_batch_no 和主流程。
互斥配套调出证券当前在互斥池时自动生成调出项，最终通过后软删原池状态。
批量整批回滚任一证券提交异常，整批日志、步骤、附件均不应半提交。
我的事宜待办普通用户只看本人 pending，管理员 1001 可见全部。
历史详情回看从历史进入必须携带 adjustBatchNo，才能看到终态批次记录。
验收时建议同步检查附件：发起上传、报告库复制、修改节点附件变更、审批通过后内部报告沉淀这四条路径。

依据文件

本页依据当前工作区代码和需求文档整理，未凭空假设接口或表。

后端核心代码

  • SecurityPoolAdjustService.java：单笔查询、校验、提交、步骤创建。
  • SecurityPoolAdjustFlowService.java：审批处理、节点推进、最终生效。
  • BatchSecurityPoolAdjustService.java：批量编排、权限、整批事务。
  • SecurityPoolQueryService.java、AdjustHistoryService.java、MyMattersService.java：查询、历史、待办。

Mapper 与表

  • SecurityPoolAdjustMapper.xml：ip_adjust_log、ip_adjust_step、ip_pool_status 读写。
  • SecurityPoolQueryMapper.xml：当前池查询口径。
  • AdjustHistoryMapper.xml：历史查询口径。
  • MyMattersMapper.xml：待办/已办事项聚合口径。
  • rrs_security_pool_adjust_schema.sql、rrs_flow_definition_schema.sql、rrs_sys_attachment_schema.sql：核心运行表、流程表、附件报告表。

前端页面

  • pages/security_pool_adjust.html：发起、校验、提交。
  • pages/batch_security_pool_adjust.html：批量选池、候选证券、批量提交。
  • pages/batch_security_pool_adjust_select.html：独立选择页，目前与主页面未形成完整回传。
  • pages/security_pool_query.html、pages/security_pool_adjust_history.html：查询与历史。
  • pages/security_pool_adjust_approve.html、pages/security_pool_adjust_detail.html、pages/my_matters.html：审核、详情、事项入口。
