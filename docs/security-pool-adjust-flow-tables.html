流程总览

按实际代码执行顺序梳理，箭头表示主要业务推进方向。

1

查询与校验

查询证券、当前池状态、投资池关系、权限及候选审批流程，判断是否允许调入或调出。

2

发起申请

每条调库明细写入 ip_adjust_log；存在附件时写入 sys_attachment。

3

创建初始步骤

读取已启用的流程版本、节点、连线和处理人配置，生成 ip_adjust_step。

4

审批流转

更新当前步骤与申请状态，并根据流程分支按需创建下一批待办步骤。

5

流程结束

通过后落地证券池状态；驳回或终止只结束申请和步骤，不改变当前池状态。

关键结论：系统没有独立的通用流程实例表。ip_adjust_log 承担申请实例，ip_adjust_step 承担步骤实例，ip_pool_status 保存最终生效的证券池状态。

主页面先定位需要调库的证券，再进入详情页加载池状态、可调池和历史流程。

请求与分页

当前所在池查询逻辑

接口同时返回“当前证券所在池”和“同发行主体所在池”，两者用途不同。

接口 POST /api/v1/securityPoolAdjust/querySecurityPoolStatus SecurityPoolAdjustService.querySecurityPoolStatus

证券当前所在池

  1. 按请求中的 securityCode 查询 ip_pool_status。
  2. 只认 audit_status='20' 且 is_deleted=0 的有效状态。
  3. 关联未删除的 ip_investment_pool。
  4. 通过 NOT EXISTS 排除仍有子节点的池，只返回叶子池。
  5. 返回目标池、池类型、批次、调库记录和入池日期。

发行主体当前所在池

  1. 先从 rrs_securityinfo 找出当前证券的发行人。
  2. 再查询同发行人的所有证券在 ip_pool_status 中的有效池状态。
  3. 同样只返回叶子池，并按目标池分组。
  4. 返回最早入池日期、最新批次号，以及主体在该池的证券数量 bondCount。

池名称回填

Service 加载全部投资池的“父级 / 子级”全路径名称映射，覆盖 SQL 返回的单级池名，使页面能够直观看到完整层级。

前端用途

securityCurrentPools 用于决定哪些池可调出、哪些池不能重复调入；issuerCurrentPools 用于展示主体整体的在池情况和辅助业务判断。

可调入 / 可调出库查询逻辑

这一段分为后端“权限基础集”和前端“方向二次过滤”，不能只看接口返回判断最终可选范围。

接口 POST /api/v1/securityPoolAdjust/queryAdjustPoolList SecurityPoolAdjustService.queryAdjustPoolList

后端：加载有权限的基础池集合

  1. 查询 ip_investment_pool 中 is_deleted=0 的未删除投资池。
  2. 管理员用户 1001 返回全量；普通用户进入权限过滤。
  3. 普通用户读取 t_sys_user_role 和 t_sys_role 获取启用角色。
  4. 从 ip_pool_permission 匹配用户或角色的“可调整”权限。
  5. 保留有权限的池及其所有祖先节点，保证前端树路径完整。

后端：补充约束和容量信息

  1. 查询 ip_pool_relation 的 in_mutex、out_mutex 关系。
  2. 按池 ID 组装 inMutexPoolIds 和 outMutexPoolIds。
  3. 汇总 ip_pool_status 中已生效且未删除的证券数量。
  4. 为每个池返回 maxCapacity 与 currentCount。

前端：生成可调出树

  1. 取当前证券有效在池 ID 集合。
  2. 只保留证券当前已经在其中的池。
  3. 同时保留这些池的父节点，用于形成可展示的树路径。
  4. 最终只有叶子节点允许勾选调出。

前端：生成可调入树

  1. 排除证券当前已经在其中的叶子池，防止重复调入。
  2. 叶子池只要未在池即可保留。
  3. 父节点仅在仍存在至少一个可调入子节点时保留。
  4. 最终只有叶子节点允许勾选调入。
当前实现说明：前端虽然分别携带 adjustDirection='in' 和 adjustDirection='out' 调用两次接口，但后端当前方法没有按该字段区分结果；两次得到相同的权限基础集合，再由前端按当前在池状态分别过滤。

调入 / 调出校验逻辑

用户点击“下一步”后，前端把勾选的叶子池提交给后端；后端返回手工项、联动项、互斥项及每项失败原因和候选流程。

接口 POST /api/v1/securityPoolAdjust/checkAdjust SecurityPoolAdjustService.checkAdjust
1. 请求校验证券代码和调库项不能为空。
2. 共享数据初始化集中加载证券、池、当前状态、关系和进行中流程。
3. 方向规则校验分别执行调入和调出规则，收集全部失败原因。
4. 匹配审批流程只为校验通过的手工项返回可选择流程。

调入校验顺序

  1. 证券是否已经到期。
  2. 是否存在 pending 的进行中调库步骤。
  3. 证券是否已经在目标池，防止重复调入。
  4. 目标池是否达到 maxCapacity 持仓上限。
  5. 若配置来源池，证券是否已在至少一个来源池。
  6. 是否命中 in_restrict 调入限制池。
  7. 本次请求是否同时调入互斥池。
  8. 是否命中 in_soft_restrict 弹性禁投池。

调出校验顺序

  1. 证券是否已经到期。
  2. 是否存在 pending 的进行中调库步骤。
  3. 证券当前是否确实在目标池中。
  4. 是否命中 out_restrict 调出限制池。
  5. 是否命中 out_mutex 调出互斥池。
  6. 本次请求是否同时调出互斥池。
  7. 是否命中 out_soft_restrict 弹性禁投池。

自动生成联动 / 互斥项

  • 调入目标池时，in_linked 关联池自动生成联动调入项。
  • 调入目标池时，若证券已在 in_mutex 池中，自动生成该池的互斥调出项。
  • 调出目标池时，out_linked 关联池自动生成联动调出项。
  • 使用“池 ID + 方向”去重；用户手工项优先于自动项。
  • 手工项失败时，同组自动项继承阻断状态。

候选流程匹配

  • 调出：使用目标池配置的标准调出流程。
  • 非信用债大库调入：使用目标池默认调入流程。
  • 信用债已在库：同父级下按 innerSort 判断上调或下调流程。
  • 信用债未在库：按“白名单 > 简易 > 默认调入”确定推荐优先级。
  • 当前白名单流程默认关闭；简易流程会校验剩余期限、同主体期限、近一年简易入池及评级条件。
当前实现说明：白名单流程入口固定返回未命中；主体评级、展望评级和担保人评级下调标识目前在共享上下文中初始化为 false，尚未接入真实评级历史查询。页面展示的是当前代码实际行为，不代表未来规则最终口径。

提交申请与审批推进逻辑

校验只判断“能不能调”；提交才创建申请和步骤，审批结束才决定是否真正改变池状态。

提交申请：五阶段

  1. 前置校验：检查证券代码、调整人和调库项等请求字段。
  2. 参数初始化：加载证券、池关系、当前状态及每个唯一流程的运行快照。
  3. 调入处理：按调库组写入手工、联动、互斥记录并绑定附件。
  4. 调出处理：同样写入申请记录，并保持同组批次关系。
  5. 后续处理：同步详情页传入的证券基础信息。

直通与人工审批

  • 无流程或流程为 start → end 时视为直通。
  • 直通调入：申请直接记为通过并新增 ip_pool_status。
  • 直通调出：申请直接记为通过并逻辑删除对应池状态。
  • 非直通：申请记为已提交，创建开始、提交人和首个待审批步骤。
  • 后续节点采用懒创建，审批到达时再写入 ip_adjust_step。

审批动作

  • 接口：POST /api/v1/securityPoolAdjustFlow/submitAdjustAudit。
  • 先校验步骤仍为待处理并确认实际处理人。
  • 更新当前步骤动作、意见、处理人和处理时间。
  • 会签节点等待全部人员处理；抢占节点处理后将其他待办标记为跳过。
  • 重新读取节点、连线和条件规则，按动作寻找下一节点。

结束落库

  • 通过：同批申请更新为通过，调入新增池状态、调出逻辑删除池状态。
  • 驳回或终止：结束申请和步骤，但不改变 ip_pool_status。
  • 通过后若存在手工上传信评附件，生成 rrs_report_in 并复制附件。
  • 修改节点重新提交时，可增删并重新绑定调库附件。

完整表清单

“条件触发”表示并非每笔调库都会访问或写入；事件表和仅用于流程设计维护的配置表不纳入运行口径。

序号 表名 用途 读写 触发场景
A. 核心业务与校验表
1ip_adjust_log 核心 调库申请明细和批次记录，保存证券、目标池、调整方向、流程及审批状态。 查询新增更新 提交调库时新增；审批时更新状态；详情、待办、历史记录和批次结束时查询。

3ip_pool_status 核心 证券当前所在投资池状态，是调库最终生效结果。 查询新增逻辑删除 校验当前池状态和容量；流程通过后，调入新增、调出逻辑删除。

7ip_pool_permission 投资池可调权限配置，支持按用户或角色授权。 查询 进入调库页面、查询可调池时判断当前用户是否有权限操作目标池。
8dict_security_type 证券类型字典及证券大类信息。 查询 证券列表展示类型名称；流程通过后生成内部报告时判断报告类型。

核心表关系

以下不是数据库物理外键，而是业务代码按照 ID、批次号和证券代码维护的逻辑关系。

申请与步骤

  • ip_adjust_log.id 对应 ip_adjust_step.adjust_log_id
  • adjust_batch_no 聚合同批手工、联动和互斥调整
  • 一个审批节点按处理人展开后可能产生多条步骤

申请与目标池

  • ip_adjust_log.target_pool_id 对应投资池 ID
  • 审批通过后按调整方向写入或逻辑删除 ip_pool_status
  • 池关系用于扩展联动与互斥调整明细

步骤与流程定义

  • ip_adjust_step.flow_node_id 指向流程节点
  • 节点归属于版本，版本归属于流程定义
  • 节点连线和条件规则共同决定下一步骤

流程结束分支

通过与未通过的最大区别，是是否真正改变证券池状态。

审批通过

更新 ip_adjust_step 和 ip_adjust_log 为完成状态；调入时新增 ip_pool_status，调出时逻辑删除对应状态。若存在手工上传的信评报告附件，再新增 rrs_report_in 并复制 sys_attachment。

驳回、拒绝或终止

更新 ip_adjust_step 和 ip_adjust_log 的处理结果及终态，不新增或删除 ip_pool_status，也不会执行审批通过后的内部报告沉淀。

口径与阅读说明

  • 完整口径共 21 张表，包含页面查询、发起校验、运行步骤、用户角色、附件和报告。
  • 若只关注流程运行与最终落库，核心是 ip_adjust_log、ip_adjust_step、ip_pool_status。
  • wf_* 表保存流程模板配置，运行过程中主要读取，不写入流程设计数据。
  • *_evt 事件表用于流程设计或投资池配置变更审计，不是本次调库实例运行的写入表。
  • wf_node_auto_config、wf_node_notify_config、wf_node_condition_config 当前未被证券池调库运行服务读取。
  • 本页依据后端 Service、Mapper XML 与 SQL 建表脚本梳理，适用于当前代码版本。
