证券池批量调整

批量选择启用叶子投资池和候选证券，逐券复用单证券调库校验，按证券分组创建调库申请并进入统一审批流程。

业务流程总览

从页面操作到数据库读写的主要路径。页面为单页两步工作台（池列表 → 选券/校验/提交），不依赖独立选择页回传。

1. 选择目标池按可调整权限查询启用叶子池，操作列调入/调出进入工作台。
2. 筛选证券步骤1：选券+材料；调入排除已在池，调出仅已在池。
3. 批量校验步骤2：逐券复用单券校验，展示结果表与材料汇总。
4. 批量提交整批事务按证券分组写申请/步骤/附件，统一 submit_time。
5. 审批生效仅 audit_status=20 落池；直通亦先锁池复核。

目标池与候选证券查询

先选有权限的启用叶子池，再按方向查询候选证券。

POST /api/v1/batchSecurityPoolAdjust/queryPoolPage · querySecurityPage

目标池查询

  • 管理员可查看全部；普通用户依据 ip_pool_permission 的 adjustable 权限过滤。
  • 只返回 status='enabled'、未删除且没有启用子节点的叶子池。
  • 汇总 ip_pool_status 的有效证券数量并回填完整池路径（按 outer_sort 等排序）。

候选证券查询

  • 目标池必须存在、启用、为叶子池且当前用户有调整权限。
  • 调入使用 NOT EXISTS 排除目标池有效状态；调出使用 EXISTS 仅保留目标池有效状态。
  • 支持证券代码、简称和市场过滤；固定排除 security_type IN ('crmw','company')。
  • 按 rrs_securityinfo.ts DESC 分页；跨页多选由前端 selectedSecurityMap 缓存。

批量校验与提交

批量层负责聚合，单券规则与证券池调库一致；页面步骤2侧重结果展示与流程选择。

POST /api/v1/batchSecurityPoolAdjust/checkAdjust · addAdjustLogWithFiles

逐券校验与页面展示

  • 每个证券转换为单证券 AdjustCheckReq，复用通用/类型特有校验、联动互斥扩展与反阻断。
  • 仅校验通过的手工项行内选择审批流程（el-select + flowOptions）；联动/互斥/关联项继承同组手工流程。
  • 步骤2表头以 chips 汇总信评报告数、其他附件数、是否放开规则、结果条数；表格无每行材料列。
  • 「调整说明」失败时「调整说明」分条展示「未通过（共 N 项）」+ 编号列表（保留全部 failReasons，检查项文案不含「无法调入」终局语气）；同组行按 adjustGroupKey 底色区分。

提交和审批

  • 整个批量请求一个事务，任一失败全部回滚；约 30 秒窗口内防重复提交（按证券集合+手工项+流程+原因意见比较）。
  • 按证券分组写 ip_adjust_log（手工标记为手动批量调整）；同次提交统一 submit_time 与共享附件上下文。
  • 目标池可配批量专用流程；未配置或 start→end 时直通，落池前锁池并动态复核。
  • releaseRules=yes 仅跳过主体内评×期限矩阵，容量/来源/硬限制等仍执行。
  • 后续审批复用证券池调库流程服务（会签/抢占/驳回/终态落库）。

关键规则与当前实现说明

  • 批量不是一条总申请：后端按证券分组，再为手工、联动、互斥、关联码明细分别保存调库日志。
  • 方向只允许 in 或 out；手工明细方向必须与本次批量方向一致。
  • 手工项必须指定审批流程；联动/互斥/关联项复用同组手工项流程与批次号。
  • 仅 audit_status=20 落地 ip_pool_status；弹性禁投为警告不阻断，与单券一致。
  • 步骤2原因/建议为整批共享文案，与单券调库页「每券可回填日志原因」场景不同。
  • 审批运行、会签/抢占、驳回和最终落库复用证券池调库流程逻辑；细节见 证券池调库 文档。
